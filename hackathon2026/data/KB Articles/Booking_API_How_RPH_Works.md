# Booking API: How RPH Works

## Purpose

**RPH (Reference Place Holder)** is a temporary identifier assigned to each passenger while creating a booking.

It allows Nitro Booking API to associate passengers with:

- Prices
- Special services
- Add-ons
- Requests
- Customer-specific data

After the booking is created, RPH is replaced with permanent customer identifiers.

---

# What is RPH?

RPH is a sequential passenger identifier.

Values range from:

```
1 ... n
```

where **n = total number of passengers**.

Example:

| Passenger | RPH |
|-----------|----:|
| John Doe | 1 |
| Maria Doe | 2 |
| Jim Doe | 3 |

---

# Booking-Level Prices

An important exception is:

```
RPH = -1
```

This represents **booking-level pricing**, rather than pricing for an individual passenger.

Examples include:

- Total booking fees
- Booking-level discounts
- Shared charges

---

# RPH Lifecycle

## Before Booking

The client assigns RPH values.

Example:

```json
{
  "customers": [
    {
      "rph": 1,
      "firstName": "John"
    },
    {
      "rph": 2,
      "firstName": "Maria"
    }
  ]
}
```

Nitro uses these identifiers to associate passenger-specific data.

---

## After Booking

Once the reservation is created:

```
RPH
        ↓
OdyCustomerRef
```

The temporary RPH is replaced by an **OdyCustomerRef**, which becomes the permanent customer identifier within Odysseus.

Some suppliers may also return a **SupplierCustomerRef**, which is the supplier's own internal passenger identifier.

---

# Customer Identifiers

## OdyCustomerRef

Generated by Odysseus.

Purpose:

- Permanent customer identifier
- Used for future booking operations
- Used by Odysseus internally

---

## SupplierCustomerRef

Generated by the cruise line.

Purpose:

- Supplier-side customer identifier
- Returned only if supported by the supplier

---

# Scenario 1 – Initial Booking

Passengers:

| Passenger | Assigned RPH |
|-----------|-------------:|
| John Doe | 1 |
| Maria Doe | 2 |

Passenger-specific pricing returned with:

- RPH 1 → John
- RPH 2 → Maria

Booking-level prices:

- RPH -1

---

# Scenario 2 – Passenger Replacement

Original booking

| Passenger | RPH |
|-----------|----:|
| John | 1 |
| Maria | 2 |

Maria cancels before sailing.

A new passenger, Jim, replaces Maria.

Result:

| Passenger | RPH |
|-----------|----:|
| John | 1 |
| Jim | 2 |

The passenger occupying that position receives the existing RPH.

> **Important:** Odysseus does not support deleting passengers after a booking is created. Passenger removals must be handled by the cruise line after booking.

---

# Scenario 3 – Adding a New Passenger

Original booking

| Passenger | RPH |
|-----------|----:|
| John | 1 |
| Maria | 2 |

Jim is added as an additional passenger.

Result

| Passenger | RPH |
|-----------|----:|
| John | 1 |
| Maria | 2 |
| Jim | 3 |

New passengers always receive the next available RPH in sequence.

---

# Business Rules

- RPH values begin at 1.
- RPH cannot exceed the passenger count.
- RPH is temporary.
- RPH is replaced by OdyCustomerRef after booking.
- Booking-level prices always use RPH = -1.
- Passenger-specific services reference passenger RPH values.

---

# Common Mistakes

## Reusing the Same RPH

Each passenger must have a unique RPH.

---

## Assuming RPH Persists Forever

RPH exists only during booking creation.

After booking, use OdyCustomerRef.

---

## Ignoring Booking-Level Prices

Do not associate:

```
RPH = -1
```

with any passenger.

These prices belong to the reservation as a whole.

---

# Best Practices

- Assign sequential RPH values starting at 1.
- Preserve RPH mappings until booking completion.
- Switch to OdyCustomerRef after booking creation.
- Handle SupplierCustomerRef only when returned by the supplier.
- Treat RPH -1 as booking-level pricing.
